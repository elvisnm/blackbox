Blackbox

AI-Powered Development Workflow
A system that connects Product Owners, Designers, and Developers through structured artifacts — so every AI conversation points at the same goal.
github.com/elvisnm/blackbox — March 2026

The Problem

Every role on our team now uses AI (Claude) in their daily work. But each role operates in isolation:

Product Owner

Talks to Claude about requirements and business logic

Designer

Talks to Claude about UI patterns and components

Developer

Talks to Claude about code and implementation

There's no shared artifact connecting these conversations to the same goal. The result:

  • Inconsistent output — the same request produces different results because intent was never captured explicitly
  • Lost decisions — after a few weeks, nobody remembers why things were built a certain way
  • Repeated context — every AI session starts from scratch because there's no persistent specification
Everyone uses AI. But without a shared path, AI conversations drift apart instead of converging toward the same goal.

The Solution

Blackbox creates a structured path where every role uses AI, but all are tied to the same goal through two connected artifacts:

Asana = The Workflow

Tracks who is doing what and when.

PO, Designer, QA, and Developer all work here. Status, assignments, notifications — this is the shared space everyone already knows.

Blueprint = The Recipe

Tells the developer (and their AI) exactly what to build, where in the code, and how to verify it works.

Lives in the project repo, right next to the source code. A technical artifact, not a workflow tracker.

The key insight: Claude is the quality gate on every side. PO's Claude writes structured Asana tickets following a strict template. Developer's Claude reads those tickets and generates technical blueprints. The template is the contract between both sides.

How It Works

The workflow is split into two worlds, connected by a bridge:

End-to-end workflow: from Asana ticket to shipped code

Step by Step

1. PO Creates a Structured Ticket

PO talks to Claude. Claude writes an Asana ticket following a strict template: Goal, Context (current behavior, desired behavior), testable Requirements, and Constraints. Claude has read-only access to the codebase, so it can reference actual pages, endpoints, and error messages — making requirements specific, not vague.

2. Designer Enriches the Ticket (optional)

If the feature needs UI work, the Designer's Claude picks up the ticket. It reads existing UI patterns in the codebase and adds screens, flows, and component choices to the ticket. Mockups are attached directly to Asana.

3. Developer Generates the Blueprint

When the ticket reaches "Ready for Development", the developer runs a single command. Claude reads the Asana ticket and generates a blueprint — a technical recipe that lives in the project repo. Then Claude researches the codebase to fill in related files, dependencies, and a phased implementation plan.

Asana Ticket → /scaffold → Blueprint → /refine → /implement → Code → PR

4. Send-backs Are Natural

If the developer finds that requirements are wrong or the UI spec doesn't work, they send the ticket back via Asana with a clear explanation. PO or Designer refines the ticket. When it comes back, the developer updates the blueprint with one command — Asana-owned sections are refreshed, technical work is preserved.

5. QA Tests Against the Ticket

QA picks up the ticket in Asana and tests against the acceptance criteria that PO wrote. No ambiguity — the requirements are testable and specific because Claude enforced the template.

Each Role Stays in Their World

Nobody has to learn a new tool. Each role works where they're already comfortable:

Role Works In Uses Claude For Output
Product Owner Asana Read codebase, write structured tickets Asana tickets
Designer Asana Research UI patterns, add specs Asana ticket enrichment + mockups
Developer Code editor + Asana Generate blueprints, implement code Blueprint + code + PR
QA Asana N/A (manual testing) Test results in Asana

PO and Designer never touch the code repository. They don't need git, a code editor, or any developer tools. They talk to Claude, and Claude writes to Asana. The structured ticket template ensures that what they produce is exactly what the developer needs.

The Contract

The Asana ticket template is the contract between all roles. It has strict rules:

  • Requirements must be testable — not "improve performance" but "page loads in under 2 seconds"
  • Current/desired behavior must be specific — reference actual pages, buttons, error messages
  • Open questions must be resolved before a ticket can move to development
  • Claude enforces the template — it warns about weak sections and blocks transitions when quality gates aren't met

Blueprint Template

Why This Matters

Higher Quality Output

Structured input produces structured output. When the AI has a specific, well-defined blueprint, the code it generates is consistent and correct.

Faster Development

Developers don't spend time interpreting vague tickets. The blueprint gives them everything they need: what to build, which files to touch, how to test it.

Persistent Knowledge

Blueprints stay in the repo as documentation. Six months later, anyone can open it and understand why a feature was built the way it was.

Zero Friction for Non-Devs

PO and Designer stay in Asana. They don't need git, a terminal, or any developer tools. Claude is their interface.

The Bigger Picture

AI is transforming how every role works. But without structure, AI conversations are ephemeral — they disappear when the session ends. Blackbox makes AI work cumulative: each role's AI contribution builds on the last, all pointing toward the same goal.

The specification is the source of truth, not the code.
Every role contributes to the specification. Code is an output.
